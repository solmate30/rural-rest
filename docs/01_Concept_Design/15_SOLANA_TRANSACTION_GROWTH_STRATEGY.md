# Solana Transaction Growth Strategy (솔라나 트랜잭션 증가 방안)
> Created: 2026-02-12 00:00
> Last Updated: 2026-02-12 00:00

본 문서는 Rural Rest가 Solana 생태계에서의 트랜잭션 수 및 사용자 활성을 늘리기 위한 방안을 정리한 것이다. RWA·온체인 수익 환원 비전(11_RWA_DAO_GOVERNANCE_VISION) 및 피치 덱의 블록체인 연동 전략과 연계되며, 규제·UX·단위 경제를 전제로 검토·추가된다.

---

## 1. 목적 및 전제

### 1.1. 목적

- Solana 네트워크 상에서의 **트랜잭션 발생량 증가**: 결제·보상·스테이킹 등 실제 사용 기반 활성화.
- **생태계·RWA 연동 기반 마련**: 향후 토큰·DAO·수익 분배가 Solana 위에서 이뤄질 경우, 이미 형성된 트랜잭션 습관·인프라 활용.
- **차별화**: 일반 법정화폐 결제만 제공하는 경쟁사 대비 "온체인 결제·혜택" 옵션을 명시적으로 제공.

### 1.2. 제약·전제

| 항목 | 내용 |
|------|------|
| **규제** | 결제·증권·전자금융 관련 법역별 검토 필수. "선택 옵션"으로 두는지 "필수"로 두는지에 따라 리스크 상이. |
| **게스트 UX** | Solana 지갑·SOL 미보유 일반 여행자 진입 장벽 고려. 온보딩·가격 예측 용이성 유지. |
| **단위 경제** | 플랫폼 수수료(10–15%) 및 호스트 수익을 해치지 않는 범위에서 할인·혜택 설계. |

---

## 2. 방안 후보

### 2.1. [검토 중] 일정 SOL 지불 + 시설 할인/혜택

| 항목 | 내용 |
|------|------|
| **안 내용** | 예: "하루 1 SOL 지불 시 해당 일 시설 사용 50% 할인 또는 기타 혜택" — 트랜잭션 발생과 혜택을 직접 연동. |
| **장점** | 인센티브가 직관적. SOL 사용 시 트랜잭션 명확히 증가. |
| **리스크** | SOL 가격 변동으로 "실제 부담 금액"이 일별로 달라져 게스트가 예측하기 어려움. 1 SOL/일이 법정화폐 기준으로 과도할 수 있음. 진입 장벽·규제 검토 필요. |
| **상태** | 검토 중. 가격 안정화 구간·최소/최대 캡 또는 "SOL 결제 옵션"으로 완화한 변형 우선 검토 권장. |

### 2.2. [우선 검토] SOL 결제 시 추가 할인 (금액 연동)

| 항목 | 내용 |
|------|------|
| **안 내용** | 숙박·체험·시설 이용을 법정화폐 대신 **SOL로 결제 시 X% 추가 할인**. 결제 금액이 서비스 가격에 연동되므로 원화·달러 기준 예측 가능. |
| **장점** | 트랜잭션은 SOL 결제 시마다 발생. 가격 변동성 노출이 "1 SOL/일 고정"보다 낮음. 기존 수수료 모델과 병행 가능. |
| **고려** | 할인율(X)과 수수료·호스트 부담 분배 시뮬레이션. 법역별 결제·환전 규제 확인. |
| **상태** | 우선 검토. 파일럿 시 소규모 노드에서 테스트 후 확대. |

### 2.3. [검토] 구독/멤버십 (월 N SOL)

| 항목 | 내용 |
|------|------|
| **안 내용** | "월 N SOL 구독 시 해당 기간 숙박·체험·시설 이용 Y% 할인" 등. 일 단위 1 SOL 부담을 구독으로 분산. |
| **장점** | 갱신 시마다 트랜잭션 발생. 반복 이용자·디지털 노마드 타겟과 맞음. |
| **고려** | N·Y 설정, 환불·해지 정책, 규제(구독형 결제·증권 여부). |
| **상태** | 검토. SOL 결제 할인(2.2) 검증 후 도입 순서 논의. |

### 2.4. [검토] 소액·다회 트랜잭션 설계

| 항목 | 내용 |
|------|------|
| **안 내용** | 단일 고액(1 SOL/일) 대신 **결제·예약 확인·리뷰 보상·체험 참가 확인** 등 여러 액션에서 소액 SOL 트랜잭션 발생하도록 설계. |
| **장점** | 동일 예산으로 트랜잭션 "횟수"를 늘릴 수 있음. 사용자 행동과 온체인 기록 연결. |
| **고려** | 가스·수수료 대비 보상 균형, 스팸·남용 방지. |
| **상태** | 검토. RWA·보상 체계(11번 문서)와 연동 설계 시 반영. |

## 3. 방안 비교 요약

| 방안 | 트랜잭션 증가 | 가격 예측 용이성 | 진입 장벽 | 수익성·규제 | 우선순위 |
|------|---------------|------------------|-----------|-------------|----------|
| 2.1 일정 SOL + 할인 | 높음 | 낮음 | 높음 | 주의 | 검토 중 |
| 2.2 SOL 결제 % 할인 | 높음 | 높음 | 중간 | 시뮬 필요 | 우선 검토 |
| 2.3 구독/멤버십 | 중간 | 중간 | 중간 | 시뮬 필요 | 검토 |
| 2.4 소액 다회 TX | 높음(횟수) | 높음 | 낮음 | 설계 필요 | 검토 |
