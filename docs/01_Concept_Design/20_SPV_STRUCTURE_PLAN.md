# 20. SPV 구조 설계 (숙소별 특수목적회사)

> Created: 2026-03-16 00:00
> Last Updated: 2026-03-28 00:00

본 문서는 Rural Rest의 RWA 토큰 발행을 위한 **숙소별 SPV(Special Purpose Vehicle, 특수목적회사) 구조**를 정의한다.

---

## 1. 기본 원칙

### 1.1. 구조 개요

**1 빈집 = 1 SPV = 1 Token Mint**

- 각 숙소마다 독립된 SPV 법인을 설립
- 각 SPV가 해당 숙소의 운영권을 보유
- 각 SPV가 독립적으로 Solana SPL Token을 발행
- RWA 토큰 = 해당 SPV 지분 (수익 배당 청구권)

### 1.2. 설계 이유

**법적 리스크 격리:**
- 한 숙소의 문제가 다른 숙소에 영향 없음
- 숙소별 자산·부채 완전 분리
- 투자자 보호 강화

**회계·세무 명확성:**
- 숙소별 수익·비용 독립 계산
- 배당 계산 투명성 확보
- 감사·공시 단순화

**증권법 대응:**
- 각 SPV가 독립적으로 증권 신고
- 투자설명서 숙소별 작성
- 규제 대응 용이

## 2. 법인 구조

### 2.1. 조직도

```
Rural Rest 주식회사 (모회사)
├─ 운영: 플랫폼, 마케팅, 고객 서비스
└─ 자회사 (SPV)
    ├─ SPV-3000 주식회사 (황오동 청송재)
    ├─ SPV-3001 주식회사 (성건동 충재댁)
    ├─ SPV-3002 주식회사 (동천동 신라숲)
    ├─ SPV-3003 주식회사 (건천읍 월성)
    └─ SPV-3004 주식회사 (안강읍 석굴재)
```

### 2.2. 모회사 (Rural Rest 주식회사)

**역할:**
- 플랫폼 운영 (예약 시스템, 결제, CS)
- 마케팅 및 브랜드 관리
- 전체 전략 수립
- SPV 통합 관리
- 지자체 협상 및 계약

**자산:**
- 플랫폼 소스코드 및 IP
- 브랜드 상표권
- 고객 데이터베이스
- 각 SPV 지분 100% 보유

**수익:**
- SPV 배당금 수령 (지분 보유 비율만큼)
- 예: SPV-3000 지분 30% 보유 시, SPV 배당의 30% 수령

### 2.3. 자회사 (각 SPV)

**명칭 규칙:**
- `SPV-[숙소ID] 주식회사`
- 예: "SPV-3000 주식회사" (황오동 청송재 운영)

**역할:**
- 해당 숙소의 운영권 보유 (지자체와 20년 계약)
- 숙소 운영 (청소, 유지보수, 게스트 응대)
- 수익 관리 및 배당 지급
- Solana SPL Token 발행 및 관리

**자산:**
- 숙소 운영권 (20년 무상 사업권)
- 운영 준비금
- 숙소 집기·비품

**수익:**
- 숙박 매출
- 체험 프로그램 수익

**비용:**
- 운영비 (청소, 공과금, 소모품 등)
- 영업이익 배분:
  - 지자체 40%
  - 마을공동체 30%
  - SPV 보유분 30% (토큰 보유자에게 배당)

## 3. SPV 설립 절차

### 3.1. 단계별 프로세스

| 단계 | 담당 | 작업 내용 | 예상 기간 | 비용 |
|------|------|----------|----------|------|
| **1. SPV 설립 준비** | 법무법인 | 정관 작성, 법인 명칭 확정 | 1주 | 50만원 |
| **2. 법인 등기** | 법무사 | 법인 설립 등기, 사업자등록 | 1~2주 | 100만원 |
| **3. 운영권 이전** | 지자체·Rural Rest | 지자체와 SPV 간 운영권 계약 체결 | 2주 | - |
| **4. 법인 계좌 개설** | 은행 | 법인 통장 개설 (수익·배당 계좌) | 1주 | - |
| **5. 감정평가** | 감정평가법인 | 운영권 가치 평가 (DCF 방식) | 2주 | 200만원 |
| **6. 증권 신고** | 금융위원회 | STO 신고 또는 소액공모 면제 신청 | 4~8주 | 500만원 |
| **7. Token Mint 생성** | 플랫폼 (Authority) | Solana SPL Token Mint 발행 | 1일 | - |

**총 소요 기간:** 약 **10~15주** (2.5~4개월)
**총 설립 비용:** 약 **850만원/SPV**

### 3.2. 5채 파일럿 기준

- **총 설립 비용:** 850만원 × 5 = **4,250만원**
- **총 소요 기간:** 병렬 진행 시 약 **4개월** (순차 진행 시 12~20개월)

## 4. 지배구조 및 권한

### 4.1. 주주 구성

**초기 설립 시:**
| 주주 | 지분 | 비고 |
|------|------|------|
| Rural Rest 주식회사 | 100% | 모회사가 전액 출자 |

**RWA 토큰 발행 후:**
| 주주 | 지분 | 비고 |
|------|------|------|
| Rural Rest 주식회사 | 30~50% | 운영 준비금 확보 후 잔여 |
| 일반 투자자 (토큰 보유자) | 50~70% | USDC로 토큰 매수 |

### 4.2. 의사결정 구조

**주요 의사결정:**
- **운영 정책:** 모회사 (Rural Rest)가 결정
- **배당 정책:** SPV 주주총회 (토큰 보유자 투표)
- **중요 계약:** 모회사 승인 필요

**DAO 거버넌스 연계:**
- RWA 토큰 = DAO Community Token
- 주요 정책은 DAO 투표로 결정
- 상세는 [18_DAO_GOVERNANCE_PLAN.md](./18_DAO_GOVERNANCE_PLAN.md) 참조

## 5. 수익 구조 및 배당

### 5.1. 독립 회계

**각 SPV 독립 관리:**
- 별도 법인 계좌
- 독립 손익계산서
- 독립 대차대조표
- 연 1회 외부 감사 (투자자 보호)

### 5.2. 수익 분배 구조

**수익 흐름 (예: SPV-3000):**

```
1박 매출 70,000원
    ↓
SPV-3000 법인 계좌 입금
    ↓
운영비 차감 (청소, 공과금, 소모품 등)
    ↓
영업이익 산정
    ↓
영업이익 배분 (40 / 30 / 30)
├─ 40% → 지자체
├─ 30% → 마을공동체
└─ 30% → SPV-3000 주주들
    ├─ Rural Rest (보유 지분만큼)
    └─ 일반 투자자 (보유 토큰만큼)
```

### 5.3. 배당 메커니즘

**월 단위 배당:**
1. 매월 말일 결산
2. 영업이익 산정
3. SPV 몫 30% 계산
4. 토큰 보유 비율대로 배당
5. Solana 프로그램 통해 자동 지급

**예시 (SPV-3000, 월 영업이익 45만원):**
- SPV 몫: 45만원 × 30% = 13.5만원
- 토큰 보유자 A (1천만 토큰 = 10%): 13.5만원 × 10% = **13,500원**
- 자동으로 지갑에 USDC 입금

## 6. 법적 고려사항

### 6.1. 자본시장법 대응

**증권 해당성:**
- RWA 토큰 = 투자계약증권 가능성 높음
- 각 SPV가 독립적으로 증권 신고 필요

**대응 전략:**
| 방안 | 조건 | 비용 |
|------|------|------|
| **소액공모 면제** | 발행가액 10억원 이하 | 낮음 |
| **증권신고서 제출** | 10억원 초과 | 높음 (수천만원) |
| **혁신금융서비스 지정** | 신청 후 승인 | 중간 |

**권장:** 초기 파일럿은 **소액공모 면제** 활용

### 6.2. 세무 전략

**법인세:**
- 각 SPV 독립 과세
- 중소기업 세액공제 가능
- 배당소득세 원천징수 의무

**부가가치세:**
- 숙박업 면세 사업자
- 플랫폼 이용료는 과세

### 6.3. 지자체 계약

**계약 당사자:**
- 기존: 지자체 ↔ Rural Rest (모회사)
- 변경: 지자체 ↔ 각 SPV

**계약 구조:**
1. 지자체 - Rural Rest 간 기본 협약
2. 지자체 - 각 SPV 간 개별 운영권 계약
3. Rural Rest - 각 SPV 간 플랫폼 이용 계약

## 7. 리스크 관리

### 7.1. SPV 독립성의 장점

**리스크 격리:**
- SPV-3000 파산해도 SPV-3001~3004는 무관
- 한 숙소 문제가 전체 프로젝트에 영향 없음

**투자자 보호:**
- 투자한 숙소만 리스크 노출
- 명확한 자산 구분
- 채권자 청구 제한

### 7.2. 운영 효율성 확보

**모회사 통합 관리:**
- 플랫폼, 마케팅, CS는 통합 운영
- 규모의 경제 달성
- 브랜드 일관성 유지

**SPV 최소 관리:**
- SPV는 수익 관리만 담당
- 실제 운영은 모회사 위탁
- 인건비 절감

## 8. 단계적 실행 계획

### Phase 1: 1개 SPV 시범 (3~6개월)

**대상:** SPV-3000 (황오동 청송재)

**목적:**
- SPV 구조 검증
- 법률·회계 프로세스 확립
- Devnet 토큰 발행 테스트

**산출물:**
- SPV 설립 매뉴얼
- 계약서 템플릿
- 회계 처리 가이드

### Phase 2: 5개 SPV 전체 확대 (6~12개월)

**병렬 진행:**
- SPV-3001 ~ SPV-3004 동시 설립
- 템플릿 활용으로 빠른 진행
- Mainnet 토큰 발행

**목표:**
- 경주 파일럿 5채 전체 RWA 완료
- 총 발행가액: 약 1.2억원 (5채 합계)

### Phase 3: 확장 (12개월~)

**다른 지역:**
- 안동, 제주 등 신규 지역 진입
- 지역별 또는 포트폴리오별 SPV 전략 수립

## 9. 온체인 SPV 구현 (Solana)

### 9.1. SPV = 매물별 전용 키페어

온체인에서 SPV는 **Solana 키페어(keypair)** 로 표현된다.

```
1 빈집 = 1 SPV = 1 Authority 키페어 = 1 Token Mint
```

- SPV 키페어의 공개키(pubkey)가 Anchor 프로그램의 `authority` 계정
- 투자금(USDC)은 `funding_vault` PDA에 에스크로
- 펀딩 조건 충족 시 → `release_funds` 실행 → USDC가 SPV 지갑으로 이체
- 운영 수익 정산 시 → `distribute_monthly_revenue` → SPV 지갑에서 `usdc_vault`로 입금
- 투자자 → `claim_dividends` → `usdc_vault`에서 지분만큼 수령

### 9.2. 키 관리 단계별 전략

| 단계 | 서명 방식 | 설명 |
|------|----------|------|
| **MVP (현재)** | 서버 사이드 | SPV 키파일을 서버 환경변수로 관리, 자동 서명 |
| **베타** | Squads 멀티시그 | Rural Rest + 지자체 대표 2-of-3 서명 필요 |
| **정식 운영** | SPV 법인 직접 서명 | HSM 또는 법인 전용 키 관리 시스템 |

**MVP 서버 사이드 방식:**
- SPV 키페어를 `AUTHORITY_KEYPAIR` 환경변수로 관리
- `initialize_property`, `release_funds`, `distribute_monthly_revenue` 모두 서버 자동 실행
- 브라우저 지갑 서명 불필요
- 먹튀 방지: 플랫폼 서버가 키를 관리 → 개인이 임의 인출 불가

**정식 운영 전환 시:**
- SPV 법인 설립 후 법인 전용 지갑으로 키 이전
- 멀티시그 도입으로 단독 서명 불가 구조 확립

### 9.3. 투자자 구매 상한

| 구분 | 구매 상한 | 의결권 상한 |
|------|----------|-----------|
| 모든 투자자 | **30%** (총 발행량 기준) | **10%** (DAO Realms 레이어) |

**변경 이유:**
- 온체인 단독으로 지갑당 10% 상한은 멀티지갑 우회가 가능하여 실효성 없음
- 구매 독점 방지보다 의결권 독점 방지가 실질적으로 중요
- 의결권 상한(10%)은 Solana Realms 거버넌스 설정으로 별도 강제

**의결권 상한 구현:**
- Solana Realms에 Community Token으로 등록 시 `maxVotingTokens` 파라미터 설정
- KYC 기반 실명 연동 전까지는 30% 단일 상한 유지
